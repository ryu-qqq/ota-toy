OTA 숙박 플랫폼의 도메인 구조를 정리해보겠습니다.

  ---
  OTA 숙박 플랫폼 도메인 구조 초안

  Bounded Context 식별

  1. 숙소 도메인 (Accommodation)

  상품 카탈로그의 핵심. "무엇을 파는가"에 해당.

  ┌──────────────────────┬───────────────────────────────────────┐
  │         개념         │                 설명                  │
  ├──────────────────────┼───────────────────────────────────────┤
  │ Property (숙소)      │ 호텔, 펜션, 모텔, 리조트 등 숙박 시설 │
  ├──────────────────────┼───────────────────────────────────────┤
  │ RoomType (객실 유형) │ 스탠다드, 디럭스, 스위트 등 객실 분류 │
  ├──────────────────────┼───────────────────────────────────────┤
  │ Amenity (편의시설)   │ 와이파이, 주차장, 조식, 수영장 등     │
  ├──────────────────────┼───────────────────────────────────────┤
  │ Image (이미지)       │ 숙소/객실 사진                        │
  ├──────────────────────┼───────────────────────────────────────┤
  │ Location (위치)      │ 주소, 좌표, 지역 분류                 │
  └──────────────────────┴───────────────────────────────────────┘

  2. 재고/요금 도메인 (Inventory & Rate)

  "언제, 얼마에, 몇 개 팔 수 있는가"에 해당. 숙소 도메인과 분리하는 이유는 변경 빈도와 생명주기가 다르기 때문. 숙소 정보는 자주 안 바뀌지만, 재고/요금은 매일 바뀐다.

  ┌──────────────────────┬────────────────────────────────────────────────┐
  │         개념         │                      설명                      │
  ├──────────────────────┼────────────────────────────────────────────────┤
  │ Inventory (재고)     │ 날짜별 객실 가용 수량. 동시성 제어의 핵심 대상 │
  ├──────────────────────┼────────────────────────────────────────────────┤
  │ Rate (요금)          │ 날짜별 객실 가격 (시즌, 요일별 차등)           │
  ├──────────────────────┼────────────────────────────────────────────────┤
  │ RatePlan (요금 정책) │ 환불가능/환불불가, 조식포함 등 요금 조건 묶음  │
  └──────────────────────┴────────────────────────────────────────────────┘

  3. 예약 도메인 (Reservation)

  고객의 구매 행위. 재고 차감과 원자적으로 처리되어야 하는 핵심 트랜잭션.

  ┌─────────────────────────────┬──────────────────────────────────┐
  │            개념             │               설명               │
  ├─────────────────────────────┼──────────────────────────────────┤
  │ Reservation (예약)          │ 예약 생성, 확정, 취소 상태 관리  │
  ├─────────────────────────────┼──────────────────────────────────┤
  │ ReservationItem (예약 항목) │ 예약 내 개별 객실/날짜 단위 항목 │
  ├─────────────────────────────┼──────────────────────────────────┤
  │ Guest (투숙객 정보)         │ 투숙객 이름, 연락처 등           │
  └─────────────────────────────┴──────────────────────────────────┘

  상태 흐름: PENDING → CONFIRMED → COMPLETED 또는 PENDING → CONFIRMED → CANCELLED

  4. 파트너 도메인 (Partner)

  "누가 숙소를 관리하는가"에 해당. Extranet의 주체.

  ┌────────────────────────────────────┬─────────────────────────────┐
  │                개념                │            설명             │
  ├────────────────────────────────────┼─────────────────────────────┤
  │ Partner (파트너)                   │ 숙소 운영 파트너 계정       │
  ├────────────────────────────────────┼─────────────────────────────┤
  │ PartnerProperty (파트너-숙소 관계) │ 파트너가 관리하는 숙소 매핑 │
  └────────────────────────────────────┴─────────────────────────────┘

  5. Supplier 통합 도메인 (Supplier)

  외부 공급자 연동. 자사 직접 등록 숙소 외에 외부 상품을 통합하는 Anti-Corruption Layer.

  ┌──────────────────────────────┬──────────────────────────────────────────┐
  │             개념             │                   설명                   │
  ├──────────────────────────────┼──────────────────────────────────────────┤
  │ Supplier (공급자)            │ 외부 숙소 공급사 정보                    │
  ├──────────────────────────────┼──────────────────────────────────────────┤
  │ SupplierAdapter              │ 외부 API 연동 + 모델 변환                │
  ├──────────────────────────────┼──────────────────────────────────────────┤
  │ SupplierProperty (통합 숙소) │ 외부 모델 → 내부 통합 모델로 변환된 숙소 │
  └──────────────────────────────┴──────────────────────────────────────────┘

  6. 검색 도메인 (Search)

  자사 숙소 + Supplier 숙소를 통합 검색하는 읽기 전용 뷰.

  ┌──────────────┬───────────────────────────────────────┐
  │     개념     │                 설명                  │
  ├──────────────┼───────────────────────────────────────┤
  │ SearchFilter │ 날짜, 인원, 가격대, 지역 등 검색 조건 │
  ├──────────────┼───────────────────────────────────────┤
  │ SearchResult │ 자사 + Supplier 통합 검색 결과        │
  └──────────────┴───────────────────────────────────────┘

  ---
  도메인 간 관계

  Partner ──소유──→ Accommodation ←──변환── Supplier
                        │
                        ↓
                 Inventory & Rate
                        │
                        ↓
                   Reservation ←──검색──── Search

  - Partner → Accommodation: 파트너가 숙소를 등록하고 관리
  - Supplier → Accommodation: 외부 공급자 상품을 내부 모델로 변환하여 통합
  - Accommodation → Inventory & Rate: 숙소/객실에 대한 날짜별 재고와 요금
  - Inventory & Rate → Reservation: 재고 차감 + 예약 생성 (원자적 처리)
  - Search → 전체: 자사 + Supplier 숙소를 통합하여 검색 결과 제공

  ---
  도메인 분리 근거

  ┌───────────────────┬────────────────────────────────────────────────────────────────┐
  │       분리        │                              이유                              │
  ├───────────────────┼────────────────────────────────────────────────────────────────┤
  │ 숙소 vs 재고/요금 │ 변경 빈도가 다름. 숙소 정보는 정적, 재고/요금은 매일 변동      │
  ├───────────────────┼────────────────────────────────────────────────────────────────┤
  │ 숙소 vs 예약      │ 생명주기가 다름. 숙소는 장기 존재, 예약은 생성→완료→종료       │
  ├───────────────────┼────────────────────────────────────────────────────────────────┤
  │ 파트너 vs 숙소    │ 책임이 다름. 파트너는 인증/권한의 주체, 숙소는 상품            │
  ├───────────────────┼────────────────────────────────────────────────────────────────┤
  │ Supplier vs 숙소  │ 데이터 소스가 다름. 외부 API 의존성을 격리 (ACL 패턴)          │
  ├───────────────────┼────────────────────────────────────────────────────────────────┤
  │ 검색 vs 나머지    │ 읽기 최적화된 별도 모델. 쓰기 도메인과 분리해야 성능 확보 가능 │
