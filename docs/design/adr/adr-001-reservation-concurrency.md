# ADR-001: 예약 생성 동시성 제어 전략

- **상태**: ACCEPTED
- **날짜**: 2026-04-06

## 맥락

동일한 객실-날짜 재고에 대해 다수의 예약 요청이 동시에 발생할 수 있다.
특히 10박 예약처럼 하나의 요청이 10개 날짜의 재고를 원자적으로 차감해야 하는 경우,
부분 차감(일부 날짜만 차감되고 나머지는 실패)이나 오버부킹이 발생하면 안 된다.

성능 테스트가 가산점 항목이므로, 정합성뿐 아니라 동시 처리량(throughput)도 설계 시 고려해야 한다.

"가상 면접 사례로 배우는 대규모 시스템 설계" 호텔 예약 시스템 챕터(p.232~262)에서
비관적 잠금, 낙관적 잠금, DB 제약 조건 3가지 방식을 비교 분석하고 있으며,
이를 기반으로 프로젝트 상황에 맞는 전략을 결정했다.

## 결정

**Redis Lua Script + DB 원자적 UPDATE 2중 구조**를 채택한다.

### 전체 흐름

```
1. 멱등키(reservationId) 확인
   → 이미 처리된 요청이면 즉시 반환 (중복 예약 방지)

2. Redis Lua Script로 다중 날짜 재고 원자적 차감
   → 모든 날짜의 재고가 충분한지 확인 후 한번에 DECRBY
   → 하나라도 부족하면 전체 롤백 (Lua 스크립트 내에서 처리)
   → 실패: InventoryExhaustedException 발생
   → Redis 장애: DB 폴백 경로로 전환

3. Reservation 도메인 객체 생성 (상태: CONFIRMED)

4. DB 원자적 UPDATE로 최종 재고 차감
   UPDATE inventory
   SET available_count = available_count - 1
   WHERE room_type_id = ? AND stock_date = ? AND available_count >= 1
   → 영향 받은 행 수 검증 (expected rows == actual rows)

5. 3~4 단계 실패 시: Redis INCRBY로 재고 복구 (보상 트랜잭션)
```

### 왜 이 방식인가

- **Redis Lua 스크립트**: Redis는 단일 스레드 모델이므로 Lua 스크립트가 원자적으로 실행된다.
  다중 날짜 재고를 하나의 Lua 호출로 검증+차감할 수 있어, 부분 차감 문제가 원천 차단된다.
- **DB 원자적 UPDATE**: `WHERE available_count >= 1` 조건으로 DB 수준에서 오버부킹을 최종 방지한다.
  Redis와 DB 사이의 정합성 갭을 메워주는 안전망 역할이다.
- **2중 구조의 이점**: Redis가 1차 게이트키퍼로 대부분의 동시 요청을 밀리초 단위로 걸러내고,
  DB는 최종 정합성만 보장하면 되므로 DB 부하가 크게 줄어든다.
- **멱등키**: 네트워크 재시도나 클라이언트 중복 요청 시 이중 예약을 방지한다.

## 대안

| 대안 | 장점 | 단점 | 비고 |
|------|------|------|------|
| 1. 비관적 잠금 (SELECT FOR UPDATE) | 구현 단순, 정합성 확실 | 동시 처리량 낮음, 데드락 위험 (다중 행 잠금 시 순서 불일치), 10박 예약 시 10행 잠금 필요 | 소규모 시스템에는 적합 |
| 2. 낙관적 잠금 (@Version) | 잠금 없이 충돌 감지, 읽기 성능 우수 | 충돌 시 재시도 필요, 동시 요청 많으면 재시도 폭증, 다중 날짜 중 일부만 충돌 시 처리 복잡 | 충돌 빈도 낮은 경우 적합 |
| 3. DB 제약 조건만 (Atomic UPDATE) | 잠금 불필요, 구현 간결, 정합성 보장 | 매 요청이 DB를 직접 타므로 동시 부하 시 DB 병목, Redis 대비 레이턴시 높음 | 중규모까지 충분 |
| 4. Redis 원자적 카운터만 (Lua Script) | 가장 빠름 (O(1)), 동시성 최고 | Redis-DB 정합성 관리 필요, Redis 장애 시 서비스 불가, 재고 데이터 유실 위험 | Redis가 SPOF가 됨 |
| **5. Redis + DB 2중 구조 (채택)** | 성능(Redis) + 정합성(DB) + 가용성(폴백) | 구현 복잡도 증가, Redis-DB 간 보상 로직 필요 | 성능 테스트 가산점 + 안정성 모두 확보 |

## 결과

### 긍정적 영향
- **성능**: Redis가 1차 게이트키퍼 역할을 하여 동시 100건 요청도 밀리초 단위로 처리 가능
- **정합성**: DB 제약 조건이 최종 방어선으로 오버부킹 원천 차단
- **가용성**: Redis 장애 시 DB 폴백 경로로 서비스 유지 가능
- **멱등성**: reservationId 유니크 제약으로 이중 예약 방지

### 트레이드오프
- **구현 복잡도**: 단순 DB만 사용하는 방식 대비 Redis Lua 스크립트, 보상 트랜잭션, 폴백 로직 등 구현량 증가
- **운영 복잡도**: Redis와 DB 간 재고 동기화 상태를 모니터링해야 함
- **Redis 의존**: Redis가 정상일 때와 장애일 때 두 가지 경로를 모두 테스트해야 함

### 구현 시 주의사항
- Redis Lua 스크립트는 단일 스크립트로 다중 키를 처리하되, 모든 키가 같은 Redis 노드에 있어야 한다 (해시 태그 사용 검토)
- 보상 트랜잭션(Redis INCRBY) 실패 시를 대비한 재고 정합성 배치 검증 로직 필요
- DB 폴백 모드에서는 낙관적 락 또는 원자적 UPDATE만으로 동작하도록 설계
