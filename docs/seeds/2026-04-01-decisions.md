# 의사결정 시드 — 2026-04-01

## Decision 1: 도메인 리서치 방법론
- **시점**: 프로젝트 시작
- **선택**: 실제 OTA 플랫폼 크롤링 기반 도메인 모델 도출
- **대안**: 
  - (A) 일반적인 상상 기반 설계 — 빠르지만 실증적 근거 없음
  - (B) 기술 블로그/문서 리서치 — 2차 자료 기반
  - (C) 실제 플랫폼 크롤링 + API 분석 — 1차 자료 기반 ✅ 선택
- **이유**: OTA 도메인을 처음 다루므로, 상상이 아닌 실제 데이터에서 도메인을 도출해야 설계 근거가 실증적. Playwright를 활용하면 빠르게 여러 플랫폼을 분석 가능
- **결과**: 도메인 초안에서 놓치고 있던 성급, 결제방식, 대실, 가격 3단구조 등 발견

## Decision 2: 크롤링 대상 플랫폼 선정
- **시점**: 리서치 시작 시
- **선택**: 해외 글로벌 OTA + 해외 아시아 OTA + 국내 OTA (3개)
- **대안**:
  - (A) 국내 OTA만 분석 — 빠르지만 시야가 좁음
  - (B) 해외만 분석 — 글로벌 표준은 파악되지만 국내 특성 놓침
  - (C) 국내 + 해외 혼합 ✅ 선택
- **이유**: 이 시스템에서 Supplier 통합이 핵심이므로, 해외 OTA가 공급자 역할을 하는 구조를 이해해야 함. 국내/해외 차이(대실, 회원가, 세금 표시 등)도 도메인 설계에 직접 영향

## Decision 3: 기술 스택 — MySQL + Redis + Testcontainers
- **시점**: 프로젝트 초기 세팅
- **선택**: H2 대신 MySQL, Testcontainers로 테스트
- **대안**:
  - (A) H2 — 간단하지만 동시성 제어(SELECT FOR UPDATE) 동작이 다름
  - (B) MySQL + Testcontainers ✅ 선택
- **이유**: 재고 차감에 비관적 락을 사용하는데, H2에서는 실제 MySQL과 동작이 달라 동시성 테스트 결과를 신뢰할 수 없음

## Decision 4: 아키텍처 — Hexagonal (Port & Adapter)
- **시점**: 프로젝트 설계 단계
- **선택**: 헥사고날 아키텍처 + 멀티모듈
- **대안**:
  - (A) 레이어드 아키텍처 — 단순하지만 도메인이 인프라에 의존
  - (B) 헥사고날 아키텍처 ✅ 선택
- **이유**: 이 시스템의 핵심이 Supplier 통합. 외부 공급자(EPS, AGODA, DOTW) 연동 시 Anti-Corruption Layer가 필수. 헥사고날의 Port/Adapter 패턴이 이 구조에 자연스러움. domain 모듈은 외부 의존성 없이 순수 Java로 유지
