# 요금 조회 캐싱 전략 설계 — 2026-04-02

## 문제
대규모 요금 조회 요청이 동시에 들어오는 상황에서 DB 부하를 어떻게 관리할 것인가.

## 구조

```
RateRule (원본 — 파트너가 Extranet에서 설정하는 요금 규칙)
    ↓ 변경 시 이벤트 발생 (RateRuleChangedEvent)
Rate (계산된 스냅샷 — 날짜별 실제 가격. RateRule + RateOverride 기반 자동 계산)
    ↓ 
Redis Cache (조회용 캐시 — 고객 검색 시 여기서 조회)
```

### 3단 레이어 역할
| 레이어 | 역할 | 변경 빈도 |
|--------|------|-----------|
| RateRule + RateOverride | 요금 규칙 원본. 파트너가 관리 | 낮음 (가끔 변경) |
| Rate | 조회용 스냅샷. 시스템이 자동 계산 | RateRule 변경 시 재계산 |
| Redis Cache | 고객 조회 캐시 | TTL 기반 자동 만료 |

### 흐름

#### 요금 설정 (Write Path)
```
1. 파트너가 Extranet에서 요금 규칙 설정
   "4/1~4/30 평일 10만, 금 12만, 토 15만"

2. RateRule 저장 → RateRuleChangedEvent 발행

3. 이벤트 리스너가 해당 기간의 Rate 스냅샷 재계산
   4/1(화)=100,000, 4/2(수)=100,000, ..., 4/5(토)=150,000

4. RateOverride가 있으면 해당 날짜 덮어쓰기
   4/5 Override: 공휴일 → 170,000

5. Redis 캐시 invalidate (해당 RatePlan + 날짜 범위)
```

#### 요금 조회 (Read Path)
```
1. 고객이 "서울, 4/2~4/4, 2박" 검색

2. Redis에서 조회 (캐시 히트 → DB 안 감)
   key: "rate:{ratePlanId}:{date}"

3. 캐시 미스 → Rate 테이블에서 조회 (이미 계산된 값)

4. 결과를 Redis에 캐시
```

## 캐시 스탬피드 방어

### 문제
캐시 TTL이 만료되는 순간, 동일 키에 대해 수백 요청이 동시에 DB로 몰림.

### 대응 전략

#### 1. 분산 락 (Redisson)
```
캐시 미스 발생
→ Redisson 분산 락 획득 시도 (key 단위)
→ 락 획득 성공: DB 조회 → 캐시 갱신 → 락 해제
→ 락 획득 실패 (다른 스레드가 이미 갱신 중): 짧은 대기 후 캐시 재조회
```
→ DB에 1번만 요청하고 나머지는 갱신된 캐시에서 읽음.

#### 2. 캐시 사전 갱신 (Cache Warming)
```
TTL 만료 전에 미리 갱신하는 방식:
- TTL = 60분
- 50분 시점에 백그라운드로 캐시 갱신
- 고객은 항상 캐시에서 읽음 (미스 자체가 거의 없음)
```

#### 3. TTL 분산 (Jittering)
```
모든 키의 TTL이 동시에 만료되면 스탬피드 발생.
TTL에 랜덤 오프셋 추가:
- 기본 TTL: 60분
- 실제 TTL: 60분 + random(0~10분)
→ 키마다 만료 시점이 분산됨
```

### 선택
- **1차**: 분산 락 (Redisson) — 동시 DB 접근 방지의 기본
- **2차**: TTL 분산 — 구현 간단하고 효과 큼
- **3차 (선택)**: 캐시 사전 갱신 — 여유가 되면 추가

## 의사결정

### 왜 3단 레이어인가
- **대안 A**: RateRule만 두고 매번 계산 → 조회마다 계산 비용 발생. 대규모 트래픽에 부적합
- **대안 B**: Rate(스냅샷)만 두고 캐시 없이 → DB 부하. 같은 날짜 같은 방 반복 조회에 비효율
- **대안 C**: 3단 레이어 (RateRule → Rate → Redis) ✅ 선택
  - 설정은 규칙 기반 (파트너 편의)
  - 조회는 미리 계산된 값 (성능)
  - 캐시로 DB 보호 (대규모 트래픽)

### 왜 분산 락인가
- **대안 A**: 단순 TTL만 → 스탬피드 방어 안 됨
- **대안 B**: 로컬 락 (synchronized) → 서버가 여러 대면 의미 없음
- **대안 C**: 분산 락 (Redisson) ✅ 선택 — 멀티 인스턴스 환경에서도 동작
