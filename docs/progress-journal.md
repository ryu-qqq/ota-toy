### 의사결정

#### 1. 도메인 리서치 방법론 — 실제 크롤링 기반
- **선택**: 실제 OTA 플랫폼 크롤링 + API 분석
- **대안**: (A) 일반적인 상상 기반 설계, (B) 기술 블로그/문서 2차 자료 기반
- **이유**: OTA 도메인을 처음 다루는 만큼, 1차 자료에서 도메인을 도출해야 설계 근거가 실증적. Playwright를 활용하면 여러 플랫폼을 빠르게 분석 가능
- **결과**: 초안에서 놓친 성급, 결제방식, 대실, 가격 3단구조 등 핵심 개념 발견

#### 2. 크롤링 대상 — 국내/해외 혼합 3개
- **선택**: 해외 글로벌 OTA + 해외 아시아 OTA + 국내 OTA
- **대안**: (A) 국내만, (B) 해외만
- **이유**: Supplier 통합 구조를 이해하려면 해외 OTA가 공급자 역할을 하는 맥락을 파악해야 함. 국내/해외 차이(대실, 회원가, 세금 표시)가 도메인 설계에 직접 영향

#### 3. 기술 스택 — MySQL + Testcontainers
- **선택**: H2 대신 MySQL, Testcontainers로 테스트
- **대안**: H2 (간단하지만 동시성 동작이 실제 RDBMS와 다름)
- **이유**: 재고 차감에 비관적 락(SELECT FOR UPDATE)을 사용하는데, H2에서는 실제 MySQL과 동작이 달라 동시성 테스트 신뢰도가 떨어짐

#### 4. 아키텍처 — Hexagonal + 멀티모듈
- **선택**: 헥사고날 아키텍처 (Port & Adapter)
- **대안**: 레이어드 아키텍처 (단순하지만 도메인이 인프라에 의존)
- **이유**: 이 시스템의 핵심은 Supplier 통합. 외부 공급자 연동 시 Anti-Corruption Layer가 필수이며, 헥사고날의 Port/Adapter 패턴이 이 구조에 자연스러움
